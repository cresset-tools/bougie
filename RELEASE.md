# Release pipeline

bougie ships binaries via `dist` (the cargo-dist successor), driven by
release-plz tags, with the bougie origin server (`origin.bougie.tools`,
a Hetzner CAX11) fronting a mirror at `releases.bougie.tools` so
`curl -LsSf https://bougie.tools/install.sh | sh` doesn't hit GitHub's
anonymous rate limits.

This doc covers the moving parts and the one-time setup. Day-to-day
the only thing that needs touching is `dist-workspace.toml` (config)
and `release-plz.toml` (versioning).

## Pipeline shape

```
            commits to main
                  │
                  ▼
        release-plz.yml (PR mode)
                  │  opens / updates a release PR
                  ▼
            user merges the PR
                  │
                  ▼
        release-plz.yml (release mode)
                  │  pushes tag `bougie-v<x.y.z>`
                  ▼
        bougie-release.yml  ← generated by `dist generate`
            │
            ├── matrix build: 7 target triples, archive + sign
            ├── upload archives + installers as workflow artifacts
            ├── create GitHub Release, attach archives + installers
            └── post-announce: publish-mirror.yml
                       │
                       ▼
            ssh + rsync as deploy@origin.bougie.tools
                       │
                       └── /srv/releases/
                              │
                              ├── github/bougie/releases/download/<tag>/<file>
                              │       (versioned, 1y immutable)
                              └── installers/bougie/latest/bougie-installer.{sh,ps1}
                                      (rolling pointer, 5m cache)
```

nginx on origin serves `/srv/releases/` at `releases.bougie.tools`.
The install one-liner `bougie.tools/install.sh` is an nginx 301 to
`releases.bougie.tools/installers/bougie/latest/bougie-installer.sh`.

The origin server already runs `index.bougie.tools` +
`blobs.bougie.tools` for the PHP distribution layer
(`cresset-tools/php-build-standalone` DISTRIBUTION.md), and the
`deploy` user that the bougie publish pipeline SSHes in as is the
same one php-build-standalone's publish pipeline uses. Both are
configured in `cresset-tools/infra` (`hosts/origin/configuration.nix`
+ `hosts/origin/nginx.nix`).

## Files

| File                                         | Purpose                                                       |
| -------------------------------------------- | ------------------------------------------------------------- |
| `dist-workspace.toml`                        | dist config: targets, installers, mirror URL, post-jobs.      |
| `release-plz.toml`                           | release-plz config: tag pattern, no GH release (dist does it).|
| `.github/workflows/release-plz.yml`          | Opens release PRs + pushes tags on merge.                     |
| `.github/workflows/bougie-release.yml`       | **Generated by `dist generate`.** Builds + uploads binaries.  |
| `.github/workflows/publish-mirror.yml`       | Rsyncs released artifacts to origin (called by bougie-release.yml).|

`bougie-release.yml` is regenerated by `dist generate` whenever
`dist-workspace.toml` changes (or `cargo-dist-version` is bumped).
**Don't edit it by hand** — your changes will be overwritten next
regeneration.

## Regenerating the release workflow

```sh
# Install dist matching the pinned `cargo-dist-version` in
# dist-workspace.toml — currently 0.32.0.
curl --proto '=https' --tlsv1.2 -LsSf \
  https://github.com/axodotdev/cargo-dist/releases/download/v0.32.0/cargo-dist-installer.sh \
  | sh

# Regenerate the workflow file in place.
dist generate

# Commit the regenerated workflow + the config change that prompted it.
git add dist-workspace.toml .github/workflows/bougie-release.yml
git commit -m "ci: update dist config + regenerate release workflow"
```

`dist plan` (no args) prints what the next release would build without
touching files — useful for sanity-checking config changes before
regenerating.

## One-time setup

### 1. Roll out the infra change

`cresset-tools/infra` already has the matching change wired up in
`hosts/origin/nginx.nix` (adds `releases.bougie.tools` + a
`bougie.tools` apex vhost with `/install.sh` + `/install.ps1`
redirects) and `hosts/origin/configuration.nix` (creates
`/srv/releases/{github,installers}` owned by `deploy`).

DNS first, then switch:

1. Point `releases.bougie.tools` and `bougie.tools` A/AAAA records at
   origin. The existing `index.bougie.tools` + `blobs.bougie.tools`
   records show the right values — copy those.
2. From the infra clone:

   ```sh
   nix run .#switch -- origin origin.bougie.tools
   ```

   First run takes a couple of minutes — ACME has to issue two new
   certificates.
3. Verify:

   ```sh
   curl -I https://releases.bougie.tools/                # 404 — host serves, empty
   curl -I https://bougie.tools/install.sh               # 301 → releases.bougie.tools/...
   ```

### 2. Generate + register a deploy SSH key

Generate a release-only key pair:

```sh
ssh-keygen -t ed25519 -C "bougie CI publish" -f /tmp/bougie-ci-publish -N ""
```

Add the public half to `deploy`'s `authorized_keys` in
`cresset-tools/infra` (`hosts/origin/configuration.nix`,
`users.users.deploy.openssh.authorizedKeys.keys`) — the placeholder
slot is already there. Switch the box (`nix run .#switch -- origin
origin.bougie.tools`) to roll the key out.

### 3. GitHub repository secrets

The publish-mirror workflow runs in the `release` environment for
secret scoping. Settings → Environments → New environment → `release`.

Add these as **environment secrets** on the `release` environment:

| Secret name           | Value                                                                                |
| --------------------- | ------------------------------------------------------------------------------------ |
| `PUBLISH_SSH_KEY`     | Contents of `/tmp/bougie-ci-publish` (the private half from step 2).                 |
| `PUBLISH_KNOWN_HOSTS` | Output of `ssh-keyscan -t ed25519,rsa origin.bougie.tools` — pins the host pubkey.   |

After adding the secrets, shred the private key locally:

```sh
shred -u /tmp/bougie-ci-publish /tmp/bougie-ci-publish.pub
```

### 4. Smoke-test the publish path

The cleanest end-to-end test is cutting a real release; release-plz
opens a PR on the next merge to `main`, and merging the release PR
fires off the whole chain.

If you want to test without bumping a version, the
`bougie-release.yml` workflow file has `on: push: tags:` only — you
can push an annotated test tag from a throwaway branch:

```sh
git tag -a bougie-v0.0.0-rc.publish-mirror -m "publish-mirror smoke test"
git push origin bougie-v0.0.0-rc.publish-mirror
# After the workflow run finishes:
git push --delete origin bougie-v0.0.0-rc.publish-mirror
git tag -d bougie-v0.0.0-rc.publish-mirror
gh release delete bougie-v0.0.0-rc.publish-mirror --yes --cleanup-tag
```

The prerelease guard (`announcement_is_prerelease` is true for any
tag with a suffix like `-rc.publish-mirror`) keeps the rolling
`installers/bougie/latest/` pointers from being clobbered by the
test release.

## Verifying the fallback

After cutting a release, smoke-test that an offline mirror still
leaves a working installer:

```sh
# Mirror path (will be used preferentially)
curl -fsSL https://bougie.tools/install.sh | bash -s -- --help

# Direct GitHub fallback — block releases.bougie.tools in /etc/hosts
# and re-run; dist's installer should fall through to
# ARTIFACT_DOWNLOAD_URLS[1] (the GitHub releases URL) and still
# install.
```
